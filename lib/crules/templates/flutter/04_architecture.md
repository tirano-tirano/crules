# アーキテクチャ設計ガイドライン

## 大原則
- **このルールズの意義**: このプルジェクトルールズは、単にAIへの指示という意味だけでなく、開発を進める上で今までの開発内容を確認するためのドキュメントという意味合いをもっています。

## システム構成とレイヤー

### プレゼンテーション層 (UI)
- **責務**
  - ユーザーインターフェースの表示
  - ユーザー入力の受け付け
  - 画面遷移の制御
  - エラー表示

- **実装方針**
  - FlutterのWidgetツリーを活用
  - 状態管理はBusiness層から提供されたProviderを利用
  - 画面ごとにディレクトリを分割
  - 共通UIコンポーネントは`common/widgets`に配置

### ビジネスロジック層 (Domain)
- **責務**
  - アプリケーションのドメインロジック
  - ユースケースの実装
  - データの変換と加工
  - バリデーション

- **実装方針**
  - ViewModelやUseCaseクラスとして実装
  - プレゼンテーション層とデータ層の仲介
  - ドメインモデルの定義
  - ビジネスルールの集約

### データ層 (Data)
- **責務**
  - データの永続化
  - API通信
  - キャッシュ管理
  - データ変換

- **実装方針**
  - リポジトリパターンを採用
  - データソースの抽象化
  - エラーハンドリング
  - キャッシュ戦略の実装

## フォルダ構成

```
lib/
├── ui/                    # プレゼンテーション層
│   ├── screens/          # 画面ごとのディレクトリ
│   │   ├── home/        # ホーム画面
│   │   ├── auth/        # 認証関連画面
│   │   └── settings/    # 設定画面
│   └── widgets/         # 共通UIコンポーネント
├── domain/               # ビジネスロジック層
│   ├── models/          # ドメインモデル
│   ├── repositories/    # リポジトリインターフェース
│   ├── usecases/       # ユースケース
│   └── viewmodels/     # ViewModel
├── data/                # データ層
│   ├── models/         # データモデル
│   ├── repositories/   # リポジトリ実装
│   ├── datasources/    # データソース
│   └── api/           # APIクライアント
└── common/             # 共通機能
    ├── constants/      # 定数定義
    ├── utils/         # ユーティリティ
    └── theme/         # テーマ設定
```

## デザイン指針・技術選定

### 状態管理
- **Riverpod**
  - グローバルな状態とScopedな状態の使い分け
  - 依存性の注入
  - テスト容易性の確保
  - パフォーマンスの最適化

### 非同期処理
- **Future/Stream**
  - async/awaitパターンの使用
  - エラーハンドリングの統一
  - キャンセル処理の実装
  - バックグラウンド処理の制御

### 依存関係
- **依存性逆転の原則**
  - インターフェースによる抽象化
  - モックによるテスト
  - 疎結合な設計
  - 拡張性の確保

### エラーハンドリング
- **Result型**
  - 成功/失敗の明示的な表現
  - エラー情報の伝播
  - ユーザーへの通知
  - ログ記録

### テスト戦略
- **テストの種類**
  - ユニットテスト（Domain層）
  - ウィジェットテスト（UI層）
  - 統合テスト（API通信）
  - E2Eテスト（主要フロー）

- **テストカバレッジ**
  - ビジネスロジック: 90%以上
  - UIコンポーネント: 80%以上
  - ユーティリティ: 100%

### パフォーマンス最適化
- **メモリ管理**
  - 不要なオブジェクトの解放
  - 画像のキャッシュ制御
  - メモリリークの防止

- **レンダリング最適化**
  - constコンストラクタの活用
  - 不要な再構築の防止
  - リスト表示の最適化

## ルールの変更に関するガイド
- **ルール更新提案**: 開発を進めていく中で、ルールに反するような開発をおこなう可能性がでてきた場合は、および、ルールに新たな内容を付け加える必要がでてきた場合は、そのまま開発を進めないでください。この場合、ユーザーの合意を得てルールを変更してください。
