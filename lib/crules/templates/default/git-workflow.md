# Git ワークフローガイド

## 大原則

- **このルールズの意義**: このプルジェクトルールズは、単に AI への指示という意味だけでなく、開発を進める上で今までの開発内容を確認するためのドキュメントという意味合いをもっています。

## コミット運用

### コミットの粒度

- **単一の責任**: 1 つのコミットは 1 つの論理的な変更のみを含むようにしてください
- **テストの整合性**: コミット時点でテストが通過している状態を維持してください
- **完結性**: コミットは独立して機能する単位としてください

### コミットメッセージ

- **形式**:

  ```
  {type}: {title}

  {body}
  ```

- **タイプ一覧**:

  - `feat`: 新機能
  - `fix`: バグ修正
  - `docs`: ドキュメント変更
  - `style`: コードスタイル変更
  - `refactor`: リファクタリング
  - `perf`: パフォーマンス改善
  - `test`: テスト関連
  - `chore`: ビルド・ツール関連

- **タイトル**:

  - 変更内容を簡潔に要約（50 文字以内）
  - 日本語で記述
  - 末尾にピリオドを付けない

- **本文**:
  - 変更の理由や詳細を記述
  - 必要に応じて箇条書きを使用
  - 関連する Issue 番号を記載

## プッシュ運用

### プッシュのタイミング

- **機能単位**: 関連する一連のコミットがまとまった時点
- **テスト完了時**: 全てのテストが通過した時点
- **レビュー準備時**: コードレビューを受ける準備が整った時点

### プッシュ前の確認事項

- テストの通過確認
- リンター警告の解消
- コンフリクトの解消
- 不要なファイルが含まれていないことの確認

## ブランチ戦略

### ブランチの種類

- `main`: 製品リリース用ブランチ
- `develop`: 開発用ブランチ
- `feature/*`: 機能開発用ブランチ
- `fix/*`: バグ修正用ブランチ
- `release/*`: リリース準備用ブランチ

### ブランチ運用ルール

- `main`ブランチは常にデプロイ可能な状態を維持
- 新機能開発は`feature`ブランチで実施
- バグ修正は`fix`ブランチで実施
- リリース準備は`release`ブランチで実施

## レビュープロセス

### プルリクエスト作成時の注意点

- 変更内容の説明を明確に記述
- レビュー範囲を適切なサイズに保つ
- 関連する Issue へのリンクを含める
- テスト結果のエビデンスを添付

### レビュー観点

- コーディング規約との整合性
- テストの十分性
- パフォーマンスへの影響
- セキュリティ上の考慮

## ルールの変更に関するガイド

- **ルール更新提案**: 開発を進めていく中で、ルールに反するような開発をおこなう可能性がでてきた場合は、および、ルールに新たな内容を付け加える必要がでてきた場合は、そのまま開発を進めないでください。この場合、ユーザーの合意を得てルールを変更してください。
